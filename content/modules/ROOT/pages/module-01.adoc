= Module 1: Architecture -- Active / Passive DR

== Overview [[overview]]

This module explains the multi-datacenter active/passive DR architecture used in this demo. There is nothing to configure here -- just review the architecture before the hands-on modules.

=== The problem

When AAP becomes mission-critical infrastructure, a database failure or datacenter outage halts automation across the enterprise. Traditional backup-and-restore approaches offer recovery times measured in hours. For platforms supporting compliance enforcement, incident response, and multi-cloud deployments, this is unacceptable.

=== The solution

Active/passive DR deploys a complete AAP stack in two datacenters:

* **DC1 (active)** -- AAP services handle all traffic; EDB Postgres primary serves all reads and writes
* **DC2 (passive)** -- AAP services are stopped; EDB Postgres standby receives streaming replication

When DC1 fails, the DC2 database is promoted and AAP services are started. Target: **RTO under 5 minutes, RPO under 5 seconds**.

=== Key points for customer conversations

* The lab collapses a production 26-VM architecture (13 per DC) into 5 VMs for demo purposes
* In production, each DC would have 8 AAP VMs (2 gateway, 2 controller, 2 hub, 2 EDA), 3 PostgreSQL VMs, plus HAProxy and Barman
* The replication and failover patterns are identical -- only the scale differs
